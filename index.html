About

Anastassios Nanos

Researcher / Systems Engineer

With over 12 years of experience in Virtualization technologies I am currently working on the lower-level parts of the stack to attack issues related to performance, scalability, power-efficiency and security in hypervisors. Previously, I was a Post-doc at CSLab, NTUA, working on bridging the gap between common HPC practices and virtualization. My research interests include I/O Virtualization, systems software for high-performance I/O in virtualized environments, systems support for heterogeneous platforms, communication architectures for clusters, and scalable storage architectures based on clusters. I have been involved in many parts of the OS software stack, including device drivers, memory management, network/block layers. I hold a Diploma in Engineering (2006) from ECE, NTUA and a PhD in Computer Engineering (2013) from NTUA. Since 2011, I serve as a co-chair of the VHPC workshop series, held in conjunction with Europar, SC, and ISC.

Memory Elasticity in modern Hypervisors

Distributed key-value stores embedded in the VMM layer.

In this project, we're working on disrupting the common case for key-value store engines deployed in VMs. Consider the following use-case: A single- or multi-tier application is deployed in the cloud and needs a secure key-value store to save and retrieve ephemeral and/or persistent data. The common case is to deploy a key-value store engine on the same virtual machine as the application, or on a separate one. This presents two major issues: (i) memory for the backing store has to be pre-allocated; (ii) the key-value store engine runs on a common OS stack (GNU/Linux for instance). We argue that by minimizing the memory footprint of the key-value engine while decoupling the backing memory store from the engine itself attacks both issues.

As an example, we extend redis, a popular key-value store, to support TMEM on KVM. Preliminary results look really promising, as our approach outperforms the common case by ~2x when the system is short on memory.

  • Further reading: UTMEM: Towards Memory Elasticity in Cloud Workloads

vAccel

Accelerating generic functions.

The vAccel framework enables workloads that execute on virtual machines to offload compute-intensive functions to backends provided by the hypervisor. To achieve this, the system presents a number of host-side accelerator functions to the guest kernel, which are backed by hardware accelerators (FPGAs, GPUs, specialized crypto engines etc.). Our main focus is a cloud environment where a user deploys an application and the system accelerates a part of it transparently (say Amazon AWS, Azure etc.). With vAccel, any application containing offload-able code exploits available host accelerators with minimal overhead.

Unikernels

Will stripping down the OS stack improve applications performance?

We are in the process of exploring the potential of unikernels in a number of application domains, using various approaches:

  • First, we are experimenting with the available unikernel frameworks to determine the feasibility of embedding an application with minimal effort into a unikernel.
  • Second, we explore supported architectures (x86 / arm64 etc.).
  • Finally, we evaluate the performance achieved with unikernel versus traditional apps deployed on a VM or a container.
